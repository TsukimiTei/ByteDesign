# ByteDesign

一句话描述你的产品：

利用 ai 输出标点符号的方式来进行 UX 与页面和功能布局设计的工具。利用 ai 输出符号快而无需实际 coding 的方式帮助设计师与 vibe coder 快速视觉化一个功能与想法

<br />

**Q: ByteDesign 的核心用户是谁？请选择主要目标用户群体：**

A: 以上都要（包括：专业 UX/UI 设计师（使用 Figma/Sketch 等工具的设计师）、Vibe Coder（用 AI 辅助编程、注重快速原型的开发者）、产品经理（需要快速验证想法和沟通需求的 PM））

<br />

**Q: 「利用 AI 输出标点符号进行布局设计」的具体交互方式是什么？**

A: 用户用自然语言描述需求，AI 用 ASCII/标点符号生成线框图（文本形式）

<br />

**Q: 生成的设计产出物最终以什么形式交付给用户？**

A: 可交互的原型（支持点击跳转等交互），并可以将布局设计信息快速同步给 coding agent

<br />

**Q: 产品的技术形态和部署方式是什么？**

A: 命令行工具（CLI，集成到开发工作流中），先做 cli作为 mvp，然后 electron

<br />

**Q: 相比现有工具（如 Figma、v0.dev、tldraw 等），ByteDesign 解决的核心痛点是什么？**

A: 成本：现有 AI 设计工具直接生成代码太重，ByteDesign 提供轻量级的中间态（符号布局）便于快速迭代。并且生成速度快。

<br />

**Q: ASCII/标点符号线框图生成后，用户如何将其转化为「可交互原型」？请选择预期的转化路径：**

A: 如果是在 cli 的框架下，就先没有这个能力**Q: CLI 版本的核心交互流程中，用户输入自然语言后，期望的输出粒度是什么级别？**

A: 看用户的需求是什么，如用户设计的是个弹窗或者卡片，那就只有卡片，如果设计的是整个页面那就是整个页面，当然也可以多页面

**Q: 用户对生成的 ASCII 线框图不满意时，如何进行迭代修改？**

A: 通过自然语言追加指令，AI 在原有基础上局部修改（如「把导航栏移到左侧」）或重新生成

<br />

**Q: MVP 阶段（CLI 版本）的成功指标，你最看重哪些？**

A: 转化准确度：生成的布局与用户意图的匹配程度（用户满意度评分）

<br />

**Q: 「将布局设计信息快速同步给 coding agent」的具体方式是什么？**

A: 如果是在 terminal 里用 claude code，是怎么交互的？包装成一个 skill 吗？

<br />

**Q: CLI 版本中，生成的 ASCII 线框图如何同步给 Coding Agent（如 Claude Code）？请选择最符合你预期的集成方式：**

A: 以上都要（包括：作为 Claude Code 的自定义 skill/command，用户在 Claude Code 中直接调用 ByteDesign、生成结构化文件（如 JSON/YAML 布局描述），Coding Agent 读取该文件作为上下文、通过剪贴板/管道（pipe）将 ASCII 输出直接传递给 Coding Agent 的 prompt）

<br />

**Q: ASCII 线框图的符号体系是否需要预定义一套标准化的组件符号库（如按钮、输入框、下拉菜单等的固定表示方式）？**

A: 否，完全由 AI 自由发挥，用最直觉的 ASCII 字符表达

<br />

**Q: CLI 工具支持的目标平台/设计场景有哪些？**

A: Web + 移动端原生 App 布局

<br />

**Q: CLI 的 AI 后端调用方式是什么？**

A: 用户自己的 claude code 或者 openrouter

<br />

**Q: 用户进行多轮迭代修改时，系统如何管理设计版本？是否需要支持版本历史回溯（如 undo/redo、查看历史版本对比）？CLI 场景下的具体交互方式是怎样的（如会话式对话、命令参数等）？**

A: 会话对话吧

<br />

**Q: CLI 工具的会话式对话中，设计版本管理的具体行为是什么？**

A: 每次生成/修改自动保存为本地文件（如 .bytedesign/ 目录），支持 `bytedesign history` 命令查看和回溯

<br />

**Q: 生成的结构化输出（同步给 Coding Agent 的 JSON/YAML）应包含哪些信息层级？**

A: 布局结构 + 样式提示 + 交互行为描述（点击事件、页面跳转、状态切换）

<br />

**Q: 对于 Web 和移动端两种场景，用户如何指定目标平台？不同平台的 ASCII 线框图是否有差异化表现（如移动端显示手机边框、底部导航栏等）？**

A: 不区分平台，统一生成通用布局，由用户自行在自然语言中说明平台特性

<br />

**Q: 当 AI 生成的 ASCII 线框图在终端中超出屏幕宽度或高度时（如复杂的多区域页面），如何处理显示和浏览？**

A: 支持终端内滚动浏览，类似 less/more 的分页查看

<br />

**Q: MVP 阶段的最小功能范围需要进一步明确：1) 第一个可用版本需要支持的核心命令有哪些（如 new/edit/export/history）？2) 是否需要支持多页面流程图（页面间跳转关系的可视化）还是 MVP 只做单页面？3) 导出格式的优先级排序是什么（纯 ASCII 文本、JSON 布局描述、Markdown、图片）？请描述你理想中 MVP 的「一次完整使用流程」。**

A: 我预期的 mvp 使用流程就是用户在 claudecode 里安装好 skill 后，打出/ 选则 bytedesign，空格，开始描述这个页面或需求是什么，就根据需求出对应设计

**Q: ByteDesign 作为 Claude Code 的 skill，用户输入描述后生成的 ASCII 线框图，其输出格式优先级是什么？MVP 阶段首先支持哪种输出？**

A: 纯 ASCII 文本线框图（直接在终端中显示）

<br />

**Q: 作为 Claude Code skill 集成时，ByteDesign 的多轮迭代修改如何与 Claude Code 的会话上下文协作？**

A: 完全依赖 Claude Code 的会话上下文，ByteDesign 不维护独立状态，每次调用都是无状态的

<br />

**Q: 用户通过** **`/bytedesign`** **skill 生成设计后，如何触发「将设计同步给 Coding Agent 实现代码」这一步？**

A: 出一个设计之后会给用户调用 claude code 的方案确认的能力，具体确认哪些问题需要对齐是根据用户的任务的。主动利用这个 plan 来询问用户是否满意，不满意则提出修改方向建议，并允许用户自行输入方案或重来。

<br />

**Q: MVP 阶段，ByteDesign skill 需要支持哪些子命令/能力？**

A: 仅支持「生成」和「迭代修改」两个核心能力，保持极简
